# VMOS Architecture Hardening

## Goal
Turn the Phase 0-8 scaffolding into contract-visible, executable VMOS behavior without claiming incomplete Linux-personality or substrate work as evidence.

## Accepted Scope
Fix the concrete review findings around process/thread semantic records, portable snapshot restore, validator lookup, clone/wait execution state, signal delivery, page fault/COW plumbing, network backend integration, epoll/futex/file-lock/rlimit/seccomp/clock behavior, and validation gates. Work in small reviewable rounds; after each implementation round, run targeted validation, review the diff, fix discovered issues, and commit.

## Current Plan
Continue in small executable rounds. Round 2 and the first two Round 3 boundaries are complete and committed. The next Round 3 sub-round should either wire the packet pump/backend into the runtime network plane without overclaiming full TCP/IP completion, continue inside the process/signal execution boundary with the non-`vfork` child execution queue required before COW fork can be claimed, or move to another bounded subsystem gap such as Futex PI.

Validation: `cargo fmt --all --check`, `git diff --check`, and `cargo check -p kernel --target x86_64-unknown-none`.

Futex PI handoff slice removes the ownerless unlock gap from the bounded `FUTEX_LOCK_PI`/`FUTEX_UNLOCK_PI` path. The futex service can now peek the highest-priority waiter without removing it, count waiters, and report the remaining highest priority after excluding the selected waiter. `UNLOCK_PI` uses that information to write the next owner tid into the futex word before waking the waiter, clears the stale `FUTEX_WAITERS` bit when no waiter remains, and transfers any remaining waiter boost from the old owner to the new owner. This is still bounded PI handoff for the current wait queue, not `WAIT_REQUEUE_PI` or a general multi-runnable PI scheduler. Validation: `cargo fmt --all --check`, `git diff --check`, `cargo test -p futex_service`, and `cargo check -p kernel --target x86_64-unknown-none`.

Live COW projection slice records successful Linux ELF COW breaks into the supervisor semantic guest-memory projection. The projection now tracks the semantic page backing each mirrored region, isolates the faulting 4KiB page before recording a COW break, bumps the page generation through `GuestMemoryManager::cow_break()`, and keeps a historical `cow-break` fault record. Both user-mode write-protection faults and syscall-side writable DMW lease materialization paths update this projection after the live page-table COW copy succeeds. This is a contract-visibility improvement for the current projection, not full live `GuestMemoryManager` ownership of every mapping. Validation: `cargo fmt --all --check`, `git diff --check`, `cargo check -p kernel --target x86_64-unknown-none`, and `cargo test -p semantic_core`. A focused host-target `cargo test -p kernel cow_break_splits_projection_page_and_bumps_generation` remains blocked by existing native-service stub drift outside this slice.

Bounded Futex PI requeue slice wires `FUTEX_WAIT_REQUEUE_PI` and `FUTEX_CMP_REQUEUE_PI` into the current wait-queue model. PI requeue waiters are registered so ordinary `FUTEX_WAKE` and non-PI requeue cannot consume them; `CMP_REQUEUE_PI` validates the compare word, migrates waiters onto the PI futex, performs ownerless handoff when possible, and updates owner boost state for owned destinations. The futex service/catalog now declare the priority/requeue exports and the `requeue` capability right they already require. This is bounded PI requeue for the existing child-runs-first wait queue, not full multi-runnable PI scheduling. Validation: `cargo fmt --all --check`, `git diff --check`, `cargo test -p futex_service`, `cargo test -p linux_syscall`, `cargo test -p contract_validate`, and `cargo check -p kernel --target x86_64-unknown-none`.

Priority-aware wait-event drain slice makes the scheduler's effective task priority affect runtime wait resolution. `drain_event_queue()` now drains a batch of pending events and applies the next event owned by the highest-priority pending wait owner, preserving FIFO order for equal priorities. This lets the existing Futex PI boost state influence wait-ready/cancel/restart application order, but it is still not a general multi-runnable scheduler or preemptive runnable-task selection. Validation: `cargo fmt --all`, `cargo fmt --all --check`, `git diff --check`, and `cargo check -p kernel --target x86_64-unknown-none`.

Wait lifecycle thread-state slice connects runtime `ThreadRuntimeStateKind::Blocked` to WaitToken registration and resolution. Registering a wait now marks the owning running thread as blocked while preserving the existing semantic wait-created state transition; resolving a wait restores only blocked/running runtime threads to semantic running state and deliberately avoids reviving stopped or dead threads. Validation: `cargo fmt --all`, `cargo fmt --all --check`, `git diff --check`, and `cargo check -p kernel --target x86_64-unknown-none`.

Current-task activation guard slice prevents the scheduler task switch helper from bypassing runtime thread state. `set_current_task()` still switches the scheduler cursor and still preserves old behavior for task ids without runtime threads, but for runtime-backed tasks it now marks the semantic task running only when the matching thread is actually running; blocked, stopped, and dead threads are not revived by task selection alone. Validation: `cargo fmt --all`, `cargo fmt --all --check`, `git diff --check`, and `cargo check -p kernel --target x86_64-unknown-none`.

Process-exit task lifecycle slice synchronizes process/thread termination with scheduler task state. `process_exit()` now marks every runtime thread in the exiting process dead and also transitions the matching semantic scheduler tasks to `Exited`, so a zombie process no longer leaves its task record looking runnable or pending. Validation: `cargo fmt --all`, `cargo fmt --all --check`, `git diff --check`, and `cargo check -p kernel --target x86_64-unknown-none`.

Blocking wait4 child-exit slice replaces the previous `ENOSYS` gap for a parent waiting on a matching running child. The Linux ELF `wait4` path now registers a `ChildExit` WaitToken, records it as `SemanticWaitKind::ChildExit`, blocks through the existing wait loop, wakes on matching child zombie transition, preserves `WNOHANG`, and returns `EINTR` through the existing signal-cancellation path when interrupted. This is child-exit wait plumbing, not full process groups/job control or a general multi-runnable scheduler. Validation: `cargo fmt --all`, `cargo check -p kernel --target x86_64-unknown-none`, `cargo fmt --all --check`, and `git diff --check`. A focused host-target `cargo test -p kernel child_exit_registration_carries_wait4_selector --no-run` attempt remains blocked by the known native-service stub drift and is not used as runner-path evidence.

Signal pause wait slice replaces `pause()`'s immediate false `EINTR` return with a real signal WaitToken. `pause()` now registers a `Signal` wait, records it as `SemanticWaitKind::Signal`, blocks through the shared wait loop, and resumes as `EINTR` only through the existing pending-signal cancellation path, after which the syscall-return checkpoint can deliver the handler/default action. This is not `sigsuspend`, full `SA_RESTART`, or alternate signal stack support. Validation: `cargo fmt --all`, `cargo check -p kernel --target x86_64-unknown-none`, `cargo fmt --all --check`, and `git diff --check`.

`rt_sigtimedwait` signal-set slice replaces the previous IO-signal-only path with bounded synchronous signal waiting. The Linux ELF path now validates `sigsetsize`, reads the caller-provided signal set, consumes already-pending matching runtime signals with optional `siginfo` output, preserves nonmatching pending IO signals, registers a `SignalSet` WaitToken for blocking waits, returns `EAGAIN` on timeout, and lets unrelated actionable signals interrupt through the existing `EINTR` cancellation path. This is not `sigsuspend`, full restart semantics, or alternate signal stack support. Validation: `cargo fmt --all --check`, `git diff --check`, `cargo check -p kernel --target x86_64-unknown-none`, and `cargo test -p linux_syscall`.

## Risks
The larger requested subsystem list is intentionally broad. The current fork/clone execution model is still child-runs-first with a suspended parent, not a general multi-runnable scheduler. `vfork` and shared-VM clone still share the active address-space representation; independent-VM `fork`/supported non-`CLONE_VM` clone now uses shared read-only COW mappings and child write-fault copy, but it is not `clone3`, thread clone, or a general scheduler COW implementation. Hidden parent fd-table references are used only while a suspended parent exists; arbitrary scheduler interleavings still need explicit cleanup paths before this can be generalized. Signal handler delivery is bounded to syscall-return checkpoints and a VMOS-private restorer frame; it is not full Linux ucontext/FPU/altstack support and does not implement `SA_RESTART`. Signal wait-interruption now exists for registered supervisor waits, and `rt_sigtimedwait` has a bounded signal-set WaitToken path, but not every ring3 polling-style syscall is represented as a blocking WaitToken yet.

Host TAP support is Linux-only, requires `/dev/net/tun` plus host permissions/configuration at runtime, records the requested MAC in backend state without configuring host link-layer address, and now has only an opt-in target-executor runtime probe rather than an always-on kernel/runtime network plane or real virtio-net queue. smoltcp packet pumping is proven for ARP-level RX/TX, TCP SYN emission, TCP established transition after a checksum-valid SYN/ACK through the stack/driver/backend pump, and a listening socket reaching established state after a remote SYN/ACK handshake. target-executor host-validation evidence records TX completion only after `PacketDeviceBackend::submit_tx` and resolves a connect-ready `SocketWritable` wait only after smoltcp reports `established`. Default evidence is still host-validation/in-memory backend evidence, not real TAP/virtio substrate evidence. Persistent TAP/virtio runtime-loop wiring remains open. Kernel connect/listen still depends on driver-service RX ingress rather than real TAP/virtio runtime input. smoltcp listen/accept is bounded to transferring one established listener socket into an accepted fd and relistening on a fresh socket; full backlog semantics, legacy-model peer identity, multi-client address tracking, and default TAP/virtio ingress remain open.
