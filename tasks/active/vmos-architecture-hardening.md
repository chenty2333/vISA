# VMOS Architecture Hardening

## Goal
Turn the Phase 0-8 scaffolding into contract-visible, executable VMOS behavior without claiming incomplete Linux-personality or substrate work as evidence.

## Accepted Scope
Fix the concrete review findings around process/thread semantic records, portable snapshot restore, validator lookup, clone/wait execution state, signal delivery, page fault/COW plumbing, network backend integration, epoll/futex/file-lock/rlimit/seccomp/clock behavior, and validation gates. Work in small reviewable rounds; after each implementation round, run targeted validation, review the diff, fix discovered issues, and commit.

## Current Plan
Continue in small executable rounds. Most original Phase 0-8 scaffolding claims now have bounded executable paths, but the remaining work is still broad. Prefer concrete gaps found by current-tree review: default network ingress/TAP/virtio runtime integration, fuller semantic memory ownership and multi-runnable scheduling, signal/job-control fidelity, generic service-path gaps, and small Linux-personality syscalls that can reuse existing contract-visible machinery without overclaiming completeness.

## Risks
`RLIMIT_MEMLOCK` is currently enforced as accounting over tracked VMOS user mappings. It does not pin physical pages, model `MLOCK_ONFAULT`/`MCL_ONFAULT`, guarantee page residency under pressure, or prove real target substrate memory-lock authority.

`RLIMIT_SIGPENDING` currently gates VMOS runtime `kill`/`tgkill` pending-signal fan-out by sender real uid and returns `EAGAIN` atomically before queuing. It does not implement Linux `sigqueue`, POSIX realtime queued value delivery, or full kernel user-struct accounting, and internal kernel-generated signals intentionally bypass this user-requested queue limit.

`RLIMIT_STACK` currently constrains the Linux ELF fixed stack mapping at exec image preparation with page-granular rounding down to the existing 8 MiB ceiling. It does not implement grow-down stack expansion, guard-page behavior, per-thread pthread stacks, or exact byte-granular overflow delivery beyond the current unmapped-page fault path.

`RLIMIT_CPU` currently charges deterministic syscall-dispatch quanta plus PIT ticks that interrupt ring3 Linux ELF execution. It queues repeated soft-limit `SIGXCPU` at each additional charged-second threshold and hard-kills on dispatch or ring3 timer tick past `rlim_max`, but it does not deliver soft-limit handlers asynchronously in the middle of user code, adjust for blocked time beyond not charging ring0 ticks, or prove a general preemptive scheduler CPU accounting model.

Actual user-context preemption/context switching, full job control/controlling-terminal semantics, full restart-block/timeout-adjusted restart semantics, full Linux ucontext/FPU signal frames, pre-`syscall` RCX/R11 recovery, and a general multi-runnable owner path remain unsupported. Futex bitset and non-PI requeue syscalls are wired. Current capability support is a bounded implemented subset with ambient, bounding-set, securebits, keepcaps `prctl` state, bounded file-capability xattr processing, and ledger transitions, not full user-namespace file-capability rootid semantics or secureexec environment behavior. `F_SETLKW` now blocks in the ring3 Linux ELF path, close/exit release locks, and dynamic native VFS locks use stable node identity across rename and hard links; the generic Linux service ABI now represents `F_SETLK`/`F_SETLKW` when `struct flock` is supplied through the linux arg buffer, and `F_GETLK` writes conflict/no-conflict data back into the same ABI-shaped flock buffer before returning success. Generic `PlanKind::Prlimit64`, `PlanKind::Getrlimit`, and `PlanKind::Setrlimit` now reach the real runtime rlimit table with explicit linux-arg-buffer `rlimit64`/x86_64 `rlimit` encoding and conservative no-capability max-raise rejection. Dynamic native VFS hard links are now modeled with shared inode records, shared content/xattrs, dynamic `nlink` reporting, and open-unlinked inode lifetime for fd-backed dynamic files; static/no-identity resources still use path fallback and cannot be hard-linked. Generic `PlanKind::RenameAt2` and `PlanKind::LinkAt` now reach the dynamic native VFS path with explicit-length generic ABI encoding, but they are not a full Linux C-string ABI and still do not make static-resource mutation/linking semantics complete. Keep false-success behavior visible as risk until replaced with real semantics.
