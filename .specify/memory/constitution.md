# vISA Constitution

## Core Principles

### I. Spec Kit First
Feature work uses Spec Kit artifacts as the active source of truth. Historical
material under `docs/archive/` is background, not an active workflow ledger.

### II. Semantic Contract Integrity
Changes that affect vISA identity, authority, capability, lifetime, generation,
trap, wait, cleanup, artifact, profile, or evidence semantics must make the
contract change explicit in the spec and validation plan.

### III. Evidence Before Claims
Do not claim semantic, portable-artifact, substrate, Linux-personality, or
performance behavior without executable evidence. Prefer the Docker validation
paths documented in `docs/DOCKER.md` when parity matters.

### IV. Complete Accepted Scope
Do not complete a feature with placeholder, toy, workaround-only, or
issue-hiding implementations. Finish the accepted scope or record the remaining
risk in the Spec Kit artifacts.

### V. Durable Documentation
Keep current decisions, plans, and tasks in Spec Kit artifacts. Use long-lived
docs only for stable project knowledge that should survive the feature.

## Governance

This constitution supersedes the previous manual task/policy workflow.
Amendments must update the relevant Spec Kit artifact and explain the migration
impact.
